## 二、整体架构概览

项目分为两大部分：

### 前端（用户看到和操作的一切）
- **3D 粒子星海场景**：流浪智子深空的全部视觉效果和交互
- **UI 界面层**：认领弹窗、蚀刻编辑面板、分享功能等
- **路由系统**：首页全景、`/s/[编号]` 个人主页直达链接

### 后端（数据存储）
- **数据库**：存储所有智子的认领状态、昵称、蚀刻内容
- **不需要用户账号系统**：认领只需昵称，用浏览器本地记忆实现"登录"

## 四、关键难点分析

### 难点1：万级粒子流畅运行（核心挑战）

**问题**：要在页面上同时渲染 2000+ 颗智子粒子、外加装饰星尘粒子，总粒子数可能达到上万甚至十万级别，还要保持 60帧流畅运行。普通方式一用就卡。

**解决方案**：使用 GPU 计算着色器（WebGPU Compute Shader），把所有粒子的运动计算交给显卡并行处理，而不是让 CPU 逐个计算。

**可行性验证**：
- Three.js 官方 Demo 已经展示了百万级粒子在 WebGPU 下流畅运行
- 自 2025 年 9 月起，Three.js r171 版本已原生支持 WebGPU，且自动降级到 WebGL 兼容老浏览器
- 参考链接中的所有粒子 Demo 都已经在用这种方案

**结论**：✅ 已有成熟方案，风险低

---

### 难点2：极端缩放——从星海全景到单颗智子

**问题**：用户可以从看到整个星海的超远视角，一路缩放到只看到一颗智子的超近视角。缩放范围极大，远看是流动粒子点，近看要变成精致的金属球体，中间过渡要自然。

**解决方案**：使用"分层渲染 + 动态切换"策略——
- 远处：智子显示为发光的点粒子（极省性能）
- 中距离：切换为小型光球
- 近处：切换为精致的 3D 金属球体，带量子纹路和环境反射

这三层之间平滑过渡，用户感知不到切换。缩放同时联动粒子流速：拉近时粒子减速，最近时几乎静止。

**可行性验证**：
- Three.js 内置 LOD（Level of Detail）系统原生支持多级别切换
- 游戏行业里这是标准做法，技术成熟

**结论**：✅ 已有成熟方案，需要精调过渡效果

---

### 难点3：鼠标粒子流交互

**问题**：鼠标移动、拖拽时要能"拨弄"粒子流，产生吸引、排斥、扰动效果，可玩性强。

**解决方案**：在 GPU 粒子运动计算中加入"鼠标力场"——鼠标位置投射到 3D 空间，作为一个引力/斥力源，附近粒子会被吸引或推开，离得远的粒子不受影响。

**可行性验证**：
- 参考链接中的 `webgpu_compute_particles_fluid` Demo 就是这个效果
- Three.js 官方的 Galaxy WebGPU Demo 展示了鼠标引力交互 + 百万粒子
- 流体粒子交互在 WebGPU 下完全可行

**结论**：✅ 已有现成参考实现，风险低

---

### 难点4：单颗智子的视觉品质

**问题**：近距离看到的单颗智子要足够精致——金属镜面球体、倒映深邃宇宙、表面流动的量子纹路、边缘光晕。这对渲染品质要求很高。

**解决方案**：使用程序化着色器生成——
- 金属镜面反射：使用环境贴图（一张宇宙星空全景图）实现倒映效果
- 量子纹路：程序化生成流动的发光线条纹路，不需要手画贴图
- 边缘光晕：后期处理的辉光效果

**可行性验证**：
- 环境贴图反射球体是 3D 渲染的基础功能，完全成熟
- 程序化纹路在 shader 中实现，性能很好
- Three.js 内置 Bloom（辉光）后期效果

**结论**：✅ 成熟方案，主要靠美术调参

---

### 难点5：认领点亮的仪式感动画

**问题**：认领瞬间要有"智子突然变亮 → 量子纹路全部点亮 → 周围星尘汇聚 → 向外扩散光波"的动画，仪式感拉满。

**解决方案**：分层组合动画——
- 智子自身：亮度从暗到亮的渐变 + 纹路从无到有的渐现
- 周围星尘：改变附近粒子的运动方向，使它们向智子汇聚
- 光波扩散：一个从智子中心向外扩展的圆环发光效果
- 整体用时间轴控制，约 2-3 秒播放完毕

**可行性验证**：
- 各效果都可以通过 shader 动画 + 粒子系统组合实现
- 参考 Codrops 的 Dreamy Particle Effect 教程有类似的粒子汇聚效果

**结论**：✅ 可行，需要较多调试来达到理想视觉效果

---

### 难点6：浏览器兼容性

**问题**：WebGPU 是新技术，目前 Chrome、Edge、Safari 26+ 支持，Firefox 还在实验阶段。如果只用 WebGPU，部分用户可能无法使用。

**解决方案**：使用 Three.js r171+ 的自动降级机制——
- 优先使用 WebGPU 渲染（性能最好）
- 浏览器不支持时自动降级到 WebGL 2（覆盖几乎所有现代浏览器）
- 降级后粒子数量适当减少，保证流畅

**可行性验证**：
- Three.js r171 起，只需一行引入代码即可自动实现降级，零配置
- 2025 年后的主流浏览器基本都支持 WebGPU

**结论**：✅ 框架已内置方案，风险极低

---

### 难点7：无账号的认领系统 + 防刷

**问题**：用户不需要注册，只输昵称就能认领。但怎么记住"这是我的智子"？怎么防止一个人疯狂认领所有智子？

**解决方案**：
- **记忆机制**：认领成功后，在浏览器本地存一个认领凭证（token），下次打开网页自动识别身份，可以编辑自己的智子
- **防刷机制**：Supabase 匿名登录 + 每个匿名身份限领 1 颗。清除浏览器数据后可以再领，但这属于高成本操作，大多数人不会这么做
- **后端数据库**：使用 Supabase 免费版，只需一张 `sophons` 表，存智子编号、认领状态、昵称、蚀刻内容、认领时间

**可行性验证**：
- Supabase 原生支持匿名登录（Anonymous Sign-In），官方文档有完整方案
- 免费版足够支撑 10 万级数据量

**结论**：✅ Supabase 原生支持，方案成熟

## 五、技术选型总结

| 领域 | 选择 | 理由 |

## 六、需要产品决策的问题

### 问题1：智子的叙事方向——三体智子 vs 微观世界

我注意到文档中有两套方向：
- `product-spec.md` 等文档：以**三体流浪智子**为叙事，用户认领的是宇宙中流浪的智子，深空粒子星海
- `contentcreater-direction.md`：提出了**微观世界**方向，认领的是微观粒子/细胞/能量簇，不是宇宙智子

两个方向的视觉风格和技术实现差别不大（都是粒子系统），但叙事语言、颜色方案、对象设计完全不同。

| 方案 | 描述 |
| --- | --- |
| **A. 三体流浪智子（深空宇宙）** | 遵循 product-spec 中的全部设定，宇宙深空背景，金属球体智子，流浪/归处叙事。三体 IP 自带认知度和传播力 |
| **B. 微观世界** | 遵循 contentcreater-direction 的方向，微观尺度，发光细胞/能量簇/粒子团，生命感/有机感叙事。更原创，但需要建立新认知 |
| **C. 融合方案：智子的微观展开** | 宏观层面是三体智子深空，但当镜头拉近到单颗智子时，智子内部展现出微观世界的视觉（量子纹路变成微观粒子结构），融合两种美感 |

**我的推荐：方案 A**
理由：产品规格文档更完整，三体 IP 认知成本低、传播力强，"流浪智子认领"这个概念一句话就能说清楚。微观世界方向缺乏一个让大众秒懂的叙事锚点。如果后续想加微观视觉，可以用方案 C 的方式在智子近景中融入。

---

### 问题2：第一阶段 Demo 的交互深度

阶段1 只做视觉 Demo，不接后端。但"可以点击智子播放展开动画"需要做到什么程度？

| 方案 | 描述 |
| --- | --- |
| **A. 纯观赏型** | 只做粒子星海全景 + 鼠标交互 + 缩放，不做点击/展开/认领动画。最快交付，约 3-5 天 |
| **B. 带假数据的完整演示** | 粒子星海 + 鼠标交互 + 缩放 + 点击智子弹出侧边栏 + 认领点亮动画 + 蚀刻展开动画，全部用假数据。约 7-10 天 |
| **C. 可录视频的完整流程** | 在 B 的基础上，确保每个动画的过渡足够丝滑，可以直接录制一段完整的视频脚本（按 owner-guide 中的传播脚本）。约 10-14 天 |

**我的推荐：方案 B**
理由：路线规划文档的阶段1目标是"可以直接录视频"，但如果只做方案 A 的纯粒子效果，视频里没有"认领"这个核心动作，传播力大减。方案 B 用假数据就能完整演示核心流程，既控制工期又保证了视频内容完整性。方案 C 的额外打磨可以在阶段2 中自然完成。

---

### 问题3：智子总数和空间分布策略

文档提到"初始放出 10000 颗智子"。但在 3D 空间中如何分布？

| 方案 | 描述 |
| --- | --- |
| **A. 固定 10000 颗，用完再生成** | 初始生成 10000 颗智子，有固定编号和固定位置。全部认领完后再自动追加。空间位置写死，不会变 |
| **B. 无限生成，按需出现** | 不预设上限，用户看到的星海是动态生成的，每个区域的智子编号根据空间坐标算出来，理论上无限。用户认领的是一个"坐标"而不是"第几号粒子" |
| **C. 分批释放** | 初始 10000 颗，每周/每月追加一批新智子，制造稀缺感和话题性 |

**我的推荐：方案 A**
理由：方案 A 最简单可靠，"10000 颗流浪智子等你认领"本身就是一个很好的传播点。固定位置方便做 `/s/[编号]` 直达链接，用户分享后别人能精确找到。方案 B 的无限生成虽然酷但增加了很多复杂度，方案 C 的分批释放可以作为运营手段后续叠加，不影响 A 的底层设计。

---

### 问题4：蚀刻编辑面板的交互形式

认领后用户要编辑自己的智子内容，这个编辑面板怎么呈现？

| 方案 | 描述 |
| --- | --- |
| **A. 侧边栏编辑** | 点击智子后右侧滑出编辑面板，背景保持3D星海但虚化。编辑完点保存，面板收回 |
| **B. 全屏二维展开** | 点击后智子播放"二维展开"动画，整个画面切换到一个全屏编辑页面，背景是智子展开的平面纹路。编辑完播放"收回"动画返回星海 |
| **C. 浮动卡片** | 智子附近弹出一个浮动卡片编辑区，不遮挡3D场景，编辑区域较小，类似气泡窗口 |

**我的推荐：方案 B**
理由：产品文档明确提到"二维展开蚀刻"的概念，方案 B 最符合这个叙事——智子真的"展开"了，用户在它的表面"刻字"。全屏编辑也给了足够的空间承载简历/链接/介绍等内容。展开和收回的动画本身就是一个视觉亮点，可以增加视频的仪式感。

---

### 问题5：移动端优先级

粒子交互在触屏上体验和桌面端差异很大（没有 hover、缩放手势不同、性能限制）。

| 方案 | 描述 |
| --- | --- |
| **A. 阶段1-2 只做桌面端** | 前两个阶段完全不考虑手机，专注把桌面端视觉效果做到极致，阶段3再适配移动端 |
| **B. 从一开始同时兼容** | 所有阶段都同时考虑桌面和手机，但可能影响开发速度和桌面端视觉上限 |
| **C. 移动端做简化版** | 桌面端做完整 3D 体验，移动端做一个简化的 2D 版本（静态背景 + 列表浏览），核心功能可用但不追求视觉效果 |

**我的推荐：方案 A**
理由：路线规划文档已经把移动端适配放在阶段3（P1），而阶段1的核心目标是"录视频传播"，视频是桌面端录制的。如果过早兼容移动端，会分散精力，拖慢视觉效果的打磨。传播视频在手机上看效果已经足够惊艳，用户被吸引后用电脑打开即可。
